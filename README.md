# Architecture Overview

Our system is designed as a scalable content recommendation pipeline that optimizes posting platform and posting time for creator content. The architecture combines FastAPI-based request handling, optimized in-memory lookup structures, lightweight worker-based batch processing, and a scoring-driven recommendation engine.

The system evaluates all possible platform-time combinations across Instagram and YouTube using creator engagement history, platform activity trends, and historical performance patterns. Recommendations are generated by selecting the combination with the highest computed engagement score.

To improve scalability and reduce latency, datasets are preprocessed into lookup dictionaries for constant-time access during recommendation generation. Incoming requests are distributed across lightweight workers that independently process assigned content batches. This design enables efficient parallel request handling while keeping worker components loosely coupled and modular.

The final recommendation determines:
- Best platform
- Best posting hour
- Whether to POST_NOW or SCHEDULE

# Worker-Based Design

The system uses lightweight worker-based processing to improve scalability and reduce bottlenecks during high request loads.

Instead of assigning workers to individual datasets or platforms, each worker independently processes a batch of content requests. This design allows workers to:
- Operate independently
- Avoid synchronization overhead
- Efficiently use shared lookup dictionaries
- Scale horizontally with increasing traffic

Each worker performs the following steps:
1. Receives a batch of content items
2. Evaluates all platform and time-slot combinations
3. Computes engagement scores
4. Selects the highest-scoring recommendation
5. Returns the processed results to the dispatcher

This coarse-grained parallel processing approach improves throughput while keeping the recommendation logic deterministic, modular, and efficient.

# Recommendation Logic

The recommendation engine computes engagement scores using:
- Creator base engagement
- Platform activity score
- Historical engagement performance

The score is calculated as:

```text
score =
base_engagement
× activity_score
× historical_engagement
```
The engine evaluates:

- Instagram + all 24 hours
- YouTube + all 24 hours

The combination with the maximum score is selected as the final recommendation.

The scheduler then determines:

- POST_NOW if the optimal time matches the current content timestamp
- SCHEDULE otherwise

# Scalability & Bottleneck Reduction

To reduce latency and improve processing efficiency, the system uses:

## Optimized Lookup Dictionaries

Datasets are converted into hash-map based lookup structures for constant-time retrieval.

Example:
```text
history_lookup[(creator, platform, content_type, hour)]
```
This removes expensive repeated DataFrame filtering operations.

## Lightweight Worker Batching

Requests are divided into batches and processed independently across workers.

## Loosely Coupled Components

The architecture separates:

- API layer
- Recommendation engine
- Worker system
- Data preprocessing
- Database layer

This modular design simplifies scaling, debugging, and parallel development.
